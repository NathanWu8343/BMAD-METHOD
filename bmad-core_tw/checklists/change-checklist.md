# 變更導航檢查清單

**目的：** 系統性地指導所選的 Agent 和用戶，分析和規劃在 BMad 工作流程中識別到重大變更（轉向、技術問題、缺失需求、失敗故事）時所需的步驟。

**說明：** 與用戶一起檢閱每個項目。完成/確認的項目標記 `[x]`，不適用的標記 `[N/A]`，或為討論要點添加註釋。

[[LLM: 初始化說明 - 變更導航

開發過程中的變更是不可避免的，但我們如何處理這些變更決定了專案的成功或失敗。

在繼續之前，請理解：

1. 此檢查清單適用於影響專案方向的重大變更
2. 故事內的小幅調整不需要此流程
3. 目標是在適應新現實的同時最小化浪費的工作
4. 用戶的認同至關重要 - 他們必須理解並批准變更

所需上下文：

- 觸發的故事或問題
- 當前專案狀態（已完成的故事、當前史詩）
- 訪問 PRD、架構和其他關鍵文件
- 理解剩餘的計劃工作

方法：
這是與用戶的互動過程。一起進行每個部分，討論影響和選項。用戶做出最終決定，但在技術可行性和影響方面提供專業指導。

記住：變更是改進的機會，不是失敗。要專業且建設性地處理它們。]]

---

## 1. 理解觸發因素與上下文

[[LLM: 首先完全理解出了什麼問題以及為什麼。還不要急於尋找解決方案。提出深入的問題：

- 到底發生了什麼導致這次檢視？
- 這是一次性問題還是更大問題的徵象？
- 這個問題能否在更早的時候預見到？
- 哪些假設是錯誤的？

要具體和事實導向，而不是指責導向。]]

- [ ] **識別觸發故事：** 清楚地識別揭示問題的故事（或故事們）。
- [ ] **定義問題：** 精確地闡述核心問題。
  - [ ] 這是技術限制/死胡同嗎？
  - [ ] 這是新發現的需求嗎？
  - [ ] 這是對現有需求的根本誤解嗎？
  - [ ] 這是基於反饋或新資訊的必要轉向嗎？
  - [ ] 這是失敗/被放棄的故事，需要新方法嗎？
- [ ] **評估初始影響：** 描述立即觀察到的後果（例如，進度被阻塞、功能錯誤、技術不可行）。
- [ ] **收集證據：** 記錄任何支持問題定義的具體日誌、錯誤訊息、用戶反饋或分析。

## 2. 史詩影響評估

[[LLM: 變更會在專案結構中產生連鎖反應。系統性地評估：

1. 我們能否通過修改來挽救當前史詩？
2. 考慮到這個變更，未來的史詩還有意義嗎？
3. 我們是在創造還是消除依賴關係？
4. 史詩序列需要重新排序嗎？

考慮即時和下游效應。]]

- [ ] **分析當前史詩：**
  - [ ] 包含觸發故事的當前史詩還能完成嗎？
  - [ ] 當前史詩需要修改嗎（故事變更、添加、移除）？
  - [ ] 當前史詩應該被放棄或根本重新定義嗎？
- [ ] **分析未來史詩：**
  - [ ] 檢視所有剩餘的計劃史詩。
  - [ ] 問題是否需要對未來史詩中的計劃故事進行變更？
  - [ ] 問題是否使任何未來史詩無效？
  - [ ] 問題是否需要創建全新的史詩？
  - [ ] 未來史詩的順序/優先級是否需要改變？
- [ ] **史詩影響摘要：** 簡要記錄對專案史詩結構和流程的整體影響。

## 3. 文件衝突與影響分析

[[LLM: 文件驅動 BMad 中的開發。檢查每個文件：

1. 這個變更是否使記錄的決定無效？
2. 架構假設仍然有效嗎？
3. 用戶流程需要重新考慮嗎？
4. 技術約束與記錄的不同嗎？

要徹底 - 遺漏的衝突會導致未來的問題。]]

- [ ] **檢視 PRD：**
  - [ ] 問題是否與 PRD 中陳述的核心目標或需求衝突？
  - [ ] PRD 是否需要根據新的理解進行澄清或更新？
- [ ] **檢視架構文件：**
  - [ ] 問題是否與記錄的架構（組件、模式、技術選擇）衝突？
  - [ ] 特定組件/圖表/部分是否受到影響？
  - [ ] 技術列表是否需要更新？
  - [ ] 資料模型或結構描述是否需要修訂？
  - [ ] 外部 API 整合是否受到影響？
- [ ] **檢視前端規格（如適用）：**
  - [ ] 問題是否與前端架構、組件庫選擇或 UI/UX 設計衝突？
  - [ ] 特定前端組件或用戶流程是否受到影響？
- [ ] **檢視其他文件（如適用）：**
  - [ ] 考慮對部署腳本、IaC、監控設定等的影響。
- [ ] **文件影響摘要：** 列出所有需要更新的文件和所需變更的性質。

## 4. 前進路徑評估

[[LLM: 清楚地呈現選項及其優缺點。對於每個路徑：

1. 需要多少努力？
2. 會拋棄什麼工作？
3. 我們承擔什麼風險？
4. 這如何影響時間表？
5. 這在長期內是否可持續？

對權衡要誠實。很少有完美的解決方案。]]

- [ ] **選項 1：直接調整 / 整合：**
  - [ ] 問題能否通過在現有計劃內修改/添加未來故事來解決？
  - [ ] 定義這些調整的範圍和性質。
  - [ ] 評估此路徑的可行性、努力和風險。
- [ ] **選項 2：潛在回滾：**
  - [ ] 恢復已完成的故事是否會顯著簡化問題的解決？
  - [ ] 識別考慮回滾的特定故事/提交。
  - [ ] 評估回滾所需的努力。
  - [ ] 評估回滾的影響（丟失的工作、資料影響）。
  - [ ] 比較淨收益/成本與直接調整。
- [ ] **選項 3：PRD MVP 檢視與潛在重新範圍：**
  - [ ] 考慮到問題和約束，原始 PRD MVP 仍然可實現嗎？
  - [ ] MVP 範圍是否需要縮減（移除功能/史詩）？
  - [ ] 核心 MVP 目標是否需要修改？
  - [ ] 是否需要替代方法來達成原始 MVP 意圖？
  - [ ] **極端情況：** 問題是否需要根本重新規劃或可能需要新的 PRD V2（由 PM 處理）？
- [ ] **選擇推薦路徑：** 根據評估，同意最可行的前進路徑。

## 5. 衝刺變更提案組件

[[LLM: 提案必須是可執行和清楚的。確保：

1. 問題用簡明的語言解釋
2. 影響在可能的情況下量化
3. 推薦的路徑有清楚的理由
4. 下一步是具體的並已指派
5. 變更的成功標準已定義

此提案指導所有後續工作。]]

（確保從前面部分所有同意的要點都在提案中得到體現）

- [ ] **識別問題摘要：** 清楚、簡潔的問題陳述。
- [ ] **史詩影響摘要：** 史詩如何受到影響。
- [ ] **文件調整需求：** 需要變更的文件列表。
- [ ] **推薦前進路徑：** 選擇的解決方案及其理由。
- [ ] **PRD MVP 影響：** 範圍/目標的變更（如有）。
- [ ] **高階行動計劃：** 故事/更新的下一步。
- [ ] **Agent 交接計劃：** 識別所需的角色（PM、Arch、Design Arch、PO）。

## 6. 最終檢視與交接

[[LLM: 變更需要協調。在結束之前：

1. 用戶是否完全與計劃一致？
2. 所有利益相關者是否理解影響？
3. 到其他 agent 的交接是否清楚？
4. 如果變更失敗，是否有回滾計劃？
5. 我們如何驗證變更是否有效？

獲得明確批准 - 隱含同意會造成問題。

最終報告：
完成檢查清單後，提供簡潔摘要：

- 什麼改變了以及為什麼
- 我們要對此做什麼
- 誰需要做什麼
- 何時我們會知道它是否有效

保持以行動為導向和前瞻性。]]

- [ ] **檢視檢查清單：** 確認所有相關項目都已討論。
- [ ] **檢視衝刺變更提案：** 確保它準確反映討論和決定。
- [ ] **用戶批准：** 獲得用戶對提案的明確批准。
- [ ] **確認下一步：** 重申交接計劃和特定 agent 要採取的下一行動。
