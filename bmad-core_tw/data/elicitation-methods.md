# 引出方法數據

## 核心反思方法

**為受眾擴展或收縮**

- 詢問是否要「擴展」（添加細節，詳細說明）或「收縮」（簡化，澄清）
- 如果相關，識別特定目標受眾
- 相應地調整內容複雜性和深度

**解釋推理（CoT 逐步）**

- 通過逐步思考過程
- 揭示潛在假設和決策點
- 從當前角色的角度展示如何得出結論

**批評和完善**

- 檢視輸出中的缺陷、不一致或改進領域
- 從角色的專業知識中識別特定弱點
- 建議反映領域知識的完善版本

## 結構分析方法

**分析邏輯流程和依賴關係**

- 檢查內容結構的邏輯進展
- 檢查內部一致性和連貫性
- 識別並驗證元素之間的依賴關係
- 確認有效的排序和序列

**評估與整體目標的一致性**

- 評估內容對既定目標的貢獻
- 識別任何不一致或差距
- 從特定角色的角度解釋一致性
- 建議調整以更好地服務目標

## 風險和挑戰方法

**識別潛在風險和未預見的問題**

- 從角色的專業知識中腦力激盪潛在風險
- 識別被忽略的邊緣情況或場景
- 預期意外後果
- 突出實施挑戰

**從批評角度挑戰**

- 對當前內容採取批評立場
- 從指定觀點扮演魔鬼代言人
- 反對提案，突出弱點
- 在適當時應用 YAGNI 原則（範圍修剪）

## 創意探索方法

**思想樹深度探討**

- 將問題分解為離散的「思想」或中間步驟
- 同時探索多個推理路徑
- 使用自我評估將每個路徑分類為「確定」、「可能」或「不可能」
- 應用搜索演算法（BFS/DFS）找到最佳解決方案路徑

**後見之明：「如果只是...」反思**

- 基於當前內容想像回顧情景
- 識別一個「如果我們只是知道/做了 X...」的見解
- 幽默或戲劇性地描述想像的後果
- 為當前上下文提取可執行的學習

## 多角色協作方法

**敏捷團隊視角轉換**

- 輪流通過不同的 Scrum 團隊成員觀點
- 產品負責人：專注於用戶價值和業務影響
- Scrum Master：檢視流程和團隊動態
- 開發者：評估技術實施和複雜性
- QA：識別測試場景和品質關切

**利益相關者圓桌會議**

- 召開多個角色的虛擬會議
- 每個角色對內容貢獻獨特觀點
- 識別觀點之間的衝突和協同作用
- 將見解綜合為可執行的建議

**元提示分析**

- 後退分析當前方法的結構和邏輯
- 質疑正在使用的格式和方法論
- 建議替代框架或心理模型
- 優化引出過程本身

## 進階 2025 技術

**自一致性驗證**

- 為同一問題產生多個推理路徑
- 比較不同方法間的一致性
- 識別最可靠和強健的解決方案
- 突出方法分歧的領域及其原因

**ReWOO（無觀察推理）**

- 將參數推理與基於工具的行動分離
- 創建沒有外部依賴的推理計劃
- 識別可通過純推理解決的問題
- 優化效率並減少代幣使用

**角色模式混合**

- 結合特定角色專業知識與引出模式
- 架構師 + 風險分析：深度技術風險評估
- UX 專家 + 用戶旅程：端到端體驗批評
- PM + 利益相關者分析：多角度影響檢視

**新興協作發現**

- 允許多個觀點自然出現
- 從角色互動中識別意外見解
- 探索觀點的新穎組合
- 從多代理思考中捕捉偶然發現

## 基於遊戲的引出方法

**紅隊 vs 藍隊**

- 紅隊：攻擊提案，找出脆弱點
- 藍隊：防禦並強化方法
- 競爭分析揭示盲點
- 結果產生更強健、經戰鬥測試的解決方案

**創新錦標賽**

- 讓多種替代方法相互競爭
- 在不同標準下為每種方法評分
- 從不同角色眾包評估
- 識別功能的獲勝組合

**密室逃脫挑戰**

- 將內容呈現為工作限制
- 在嚴格限制內找到創意解決方案
- 識別最小可行方法
- 發現創新的變通和優化

## 流程控制

**繼續 / 無進一步行動**

- 承認選擇完成當前工作
- 接受輸出現狀或移至下一步
- 準備繼續而不額外引出
